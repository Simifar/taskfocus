# ЧАСТЬ I. СТРУКТУРА ДИПЛОМА: ЧТО И КАК ПИСАТЬ

## 1. Введение (5-7 страниц)

### 1.1. Актуальность проблемы
**Что писать:**
- Статистика СДВГ: 2.5-4% взрослых, 5-7% детей (по данным ВОЗ)
- Проблема: традиционные task-менеджеры не учитывают нейробиологические особенности
- Цифровая трансформация и рост дистанционной работы усугубляют проблему

**Ключевые источники:**
- Asherson et al. (2012) — executive dysfunction и продуктивность
- Ptáček et al. (2019) — "time blindness" и когнитивная перегрузка
- Turjeman-Levi et al. (2024) — связь СДВГ и выгорания

### 1.2. Цели и задачи
**Цель:** Разработать программный продукт, повышающий продуктивность пользователей с СДВГ через адаптивные UX-паттерны

**Задачи:**
1. Провести анализ существующих решений и их ограничений
2. Разработать архитектуру адаптивного task-менеджера
3. Реализовать ключевые модули (ограничение задач, учёт энергии, мягкие дедлайны)
4. Провести тестирование и оценить эффективность

### 1.3. Объект и предмет исследования
- **Объект:** Процесс управления задачами людьми с СДВГ
- **Предмет:** Методы и алгоритмы адаптации task-менеджеров под когнитивные особенности СДВГ

### 1.4. Методы исследования
- Аналитический (обзор литературы и существующих решений)
- Проектирование (UML, ER-диаграммы)
- Экспериментальный (пользовательское тестирование)
- Сравнительный (до/после внедрения)

### 1.5. Научная новизна
- Систематизация UX-паттернов для СДВГ-пользователей
- Алгоритм ограничения активных задач (максимум 3)
- Модель учёта энергетического состояния пользователя

### 1.6. Практическая ценность
- Готовый программный продукт
- Методика адаптации интерфейсов для нейроинклюзивности
- Рекомендации для разработчиков

## 2. Аналитический обзор (15-20 страниц)

### 2.1. Современные task-менеджеры

**Структура анализа каждого инструмента:**
| Критерий | Описание |
|----------|----------|
| Назначение | Основная функция |
| Целевая аудитория | Для кого разработан |
| Ключевые функции | Что умеет |
| Ограничения для СДВГ | Почему не подходит |

**Инструменты для анализа:**
1. **Todoist** — классический to-do, перегруз списками
2. **Trello** — Kanban, сложность приоритизации
3. **Notion** — гибкость, но когнитивная перегрузка
4. **Microsoft To Do** — простота, но нет адаптации
5. **Focusmate** — body doubling (хороший пример)

**Вывод:** Классические инструменты предполагают стабильное внимание и executive function, которых нет у СДВГ-пользователей

### 2.2. Нейробиологические основы СДВГ

**Что включить:**
- Различия в дофаминергической системе
- "Стена ужаса" (wall of awful) при инициации задач
- Time blindness — искажение восприятия времени
- Рабочая память и когнитивная перегрузка

**Источники:**
- ADD Resource Center (2025) — нейронаука task management
- Sosnoski (2022) — мозговые механизмы восприятия времени

### 2.3. Теория когнитивной нагрузки (Cognitive Load Theory)

**Ключевые концепции:**
- Внутренняя нагрузка (intrinsic) — сложность задачи
- Внешняя нагрузка (extraneous) — шум интерфейса
- Релевантная нагрузка (germane) — полезная обработка

**Применение к дизайну:**
- Минимизация внешней нагрузки через простой UI
- Разбиение сложных задач (chunking)
- Визуальные подсказки для снижения нагрузки на рабочую память

### 2.4. UX-паттерны для нейродивергентных пользователей

**Проверенные паттерны:**

| Паттерн | Реализация | Источник |
|---------|------------|----------|
| Пошаговые процессы | Разбиение на микро-шаги | PrintMag (2025) |
| Визуальное подтверждение | Статусы, прогресс-бары | TGTH.pl (2025) |
| Контроль над темпом | Без тайм-лимитов, сохранение прогресса | TGTH.pl (2025) |
| Снижение визуального шума | Минимализм, фокус-режим | LinkedIn/Babich (2025) |
| Body doubling | Присутствие "напарника" | Focusmate.com |

## 4. Проектирование системы (15-20 страниц)

### 4.1. Архитектура: Clean Architecture / Onion Architecture

```
┌─────────────────────────────────────┐
│           Presentation (WPF)        │
│  - Views, ViewModels, Converters    │
├─────────────────────────────────────┤
│         Application Layer           │
│  - Services, DTOs, Interfaces       │
├─────────────────────────────────────┤
│          Domain Layer               │
│  - Entities, Business Rules         │
├─────────────────────────────────────┤
│       Infrastructure Layer          │
│  - Repositories, EF Core, DB        │
└─────────────────────────────────────┘
```

**Обоснование:**
- Чистая архитектура обеспечивает тестируемость
- Легко добавить Web API позже (тот же Domain)
- Разделение ответственности

### 4.2. ER-диаграмма

**Сущности:**

```
[User]
- Id (PK)
- Username
- Email
- CreatedAt

[Task]
- Id (PK)
- UserId (FK)
- Title
- Description
- Priority (Low/Medium/High)
- Status (Active/Completed/Archived)
- EnergyLevel (1-5)
- DueDateStart (мягкий дедлайн - от)
- DueDateEnd (мягкий дедлайн - до)
- ParentTaskId (FK, nullable)
- CreatedAt
- CompletedAt

[TaskHistory]
- Id (PK)
- TaskId (FK)
- Action (Created/Updated/Completed)
- Timestamp
- Details
```

### 4.3. Диаграмма компонентов

**Описание:**
- MainWindow — главное окно
- TaskListView — список задач
- TaskEditView — создание/редактирование
- DashboardView — статистика
- TaskService — бизнес-логика
- TaskRepository — доступ к данным
- ValidationService — проверка ограничений

### 4.4. Ключевые алгоритмы

**Алгоритм проверки лимита задач:**
```csharp
public bool CanActivateTask(int userId)
{
    var activeCount = _taskRepository
        .Count(t => t.UserId == userId && t.Status == TaskStatus.Active);
    return activeCount < 3;
}
```

**Алгоритм фильтрации по энергии:**
```csharp
public IEnumerable<Task> GetTasksByEnergy(int userId, int currentEnergy)
{
    return _taskRepository
        .Where(t => t.UserId == userId && t.EnergyLevel <= currentEnergy)
        .OrderBy(t => t.EnergyLevel);
}
```

## Возможные вопросы на защите

1. **Почему именно 3 задачи?**  
   *Ответ:* Исследования показывают, что рабочая память ограничена 3-4 объектами (rule of three). Это снижает когнитивную нагрузку.

2. **Как вы проверяли эффективность?**  
   *Ответ:* Pre-post дизайн: неделя с обычным to-do, неделя с TaskFocus. Метрики: количество выполненных задач, субъективная оценка.

3. **Почему WPF, а не Web?**  
   *Ответ:* MVP фокус на десктопе для глубокой работы. Web — в планах. WPF даёт богатый UI без сложности Web.

4. **Как учитывается энергия?**  
   *Ответ:* Пользователь оценивает свой уровень (1-5), задачи помечаются требуемой энергией. Система предлагает задачи, соответствующие текущему состоянию.

5. **Что такое "мягкие дедлайны"?**  
   *Ответ:* Диапазон дат вместо точной даты. Снижает тревожность при СДВГ, где time blindness мешает точному планированию.

6. **Как масштабировать на команду?**  
   *Ответ:* Добавить сущность Team, права доступа, совместные задачи. Body doubling для команд — перспектива.

7. **Какие ограничения у вашего решения?**  
   *Ответ:* Нет мобильной версии, нет синхронизации, ограниченная выборка при тестировании.
